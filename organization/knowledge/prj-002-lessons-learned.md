# PRJ-002 プロジェクト振り返り（KPT）

## プロジェクト概要
- **案件**: PDF帳票OCR/VLMデータ自動抽出システム
- **期間**: 2026年3月
- **技術スタック**: Next.js + Supabase + Vercel + AWS S3

## Keep（継続すべきこと）

### 1. SWRキャッシュパターン + 同期初期値
- `cachedFetch()` でTTL内はAPI呼び出しゼロ、TTL切れは再取得
- `cacheGetSync()` で `useState` の初期値にキャッシュデータを注入
- **効果**: メニュー遷移時のローディングを完全排除
- **横展開**: 全プロジェクトの標準パターンとして採用推奨

### 2. サーバーサイドプロキシパターン（外部API連携）
- ブラウザ→API Route→外部サービスの3層構成
- **適用例**: S3, 外部AI API, Webhook送信
- **効果**: CORS問題の回避、認証情報のサーバーサイド保持、監査ログ取得

### 3. インクリメンタルマイグレーション戦略
- Phase 1〜10の段階的マイグレーション
- 各PhaseにDEPLOYMENT.mdで適用手順を記載
- **効果**: 本番環境への段階的適用が可能、ロールバックが容易

### 4. コードレビューの厳格化
- CRITICAL/HIGH/MEDIUM/LOWの4段階
- CRITICAL/HIGHはマージブロック
- **効果**: セキュリティ問題（認証情報漏洩、権限不備）を本番前に検出

### 5. テンプレート配布（マスタ→テナントコピー）パターン
- 共有テンプレート → テナントにIDリマッピング付きコピー
- 配布記録の管理でバージョン追跡
- **横展開**: マスタデータ配布が必要なマルチテナントシステム全般

## Problem（問題があったこと）

### 1. PDFデータURLの二重プレフィックス
- **原因**: APIが`data:application/pdf;base64,...`を返し、ビューアーが再度プレフィックスを追加
- **教訓**: データ形式の契約（raw base64 vs data URL）を明確に定義する
- **対策**: ビューアー側でプレフィックス有無をチェックする防御的コーディング

### 2. UNIQUE制約とマルチテナントの不整合
- **原因**: `physical_name UNIQUE` が単体制約で、テナント間で同名テーブルが作れない
- **教訓**: マルチテナント化時にUNIQUE制約は必ず `(カラム, tenant_id)` の複合にする
- **チェックリスト項目追加**: マルチテナント対応時にUNIQUE制約を全件レビュー

### 3. localStorage → Supabase移行時のデータ不整合
- **原因**: 配布記録がlocalStorageのみで、Supabase移行後に孤立データが発生
- **教訓**: ストレージ層のインターフェースをasync化し、実装を差し替え可能にする
- **パターン**: 最初からasyncインターフェースで設計する

### 4. N+1クエリパターン
- **発生箇所**: テナント一覧でテナント数分のカウントAPIを呼び出し
- **教訓**: 一覧画面の設計時にN+1を意識し、バッチ取得またはキャッシュで対応
- **対策**: SWRキャッシュ + プリフェッチで体感速度を改善

### 5. setDistributionFlagsのlocalStorage直接操作
- **原因**: Supabase対応時にAPI経由に変更すべき箇所が残存
- **教訓**: ストレージ抽象化レイヤーを必ず経由する。直接localStorageを操作しない

## Try（次回試みること）

### 1. 初期設計でのストレージ抽象化
- 最初からasyncインターフェースで設計
- localStorage実装とSupabase実装を同時に用意
- テストはインターフェース経由のみで行う

### 2. React Queryの早期導入
- SWRキャッシュを自前実装したが、React Query/SWRライブラリを使えば同等の機能が少ないコードで実現可能
- **推奨**: 次回プロジェクトではReact Queryを標準採用

### 3. RLS設計の初期段階での確定
- マイグレーションPhaseが増えるたびにRLSポリシーを追加する形になった
- **改善**: schema.sql作成時にRLSポリシーも含めて設計する

### 4. エラーハンドリングの統一フレームワーク
- `apiError()` ユーティリティは後から追加した
- **改善**: プロジェクト初期に構造化ログ+エラーハンドリングを標準化

### 5. セキュリティヘッダーの標準テンプレート
- `next.config.ts` のセキュリティヘッダーは全プロジェクト共通
- **テンプレート化**: プロジェクトセットアップチェックリストに含める

## 横展開可能な技術パターン

### パターン1: マルチテナントデータアクセス層
```
TenantStorageProvider (React Context)
  ├── createSupabaseXxxStorage(tenantId) → テナントスコープのデータアクセス
  ├── SWRキャッシュ(storage-cache.ts) → テナントID付きキーイング
  └── プリフェッチ(requestIdleCallback) → ログイン後に全データ事前取得
```

### パターン2: 外部サービスプロキシ
```
ブラウザ → /api/data/xxx → サーバーサイドで認証情報解決 → 外部API
  ├── configId方式: DBから認証情報を取得（本番用）
  └── config直接方式: 接続テスト用（設定保存前）
```

### パターン3: テンプレート配布システム
```
管理コンソール(共有テンプレート) → distributeToTenant()
  ├── テーブル定義コピー（IDリマッピング）
  ├── カラム定義コピー（DB生成IDで再構築）
  ├── テンプレートコピー（PDFデータ含む）
  ├── フィールドマッピングコピー
  ├── isShared/sharedByフラグ設定
  └── 配布記録保存
```

### パターン4: パフォーマンス最適化の3段階
```
Stage 1: SWRキャッシュ（TTL + staleデータ即返却）
Stage 2: useState初期値にcacheGetSync()で同期注入
Stage 3: ログイン後にrequestIdleCallbackで全データプリフェッチ
```
