# 品質ゲート定義

## 概要
各Phaseの移行には品質ゲートの通過が必須です。
品質ゲートを通過せずにPhaseを進めることは禁止されています。

## Phase移行条件

### Gate 1: 受付 → 調査
**判定者**: 秘書 + CEO
- [ ] brief.md の必須項目が全て埋まっている
- [ ] クライアント名・連絡先が確認されている
- [ ] 案件の概要が明確に記述されている
- [ ] 希望納期が確認されている
- [ ] 予算感が確認されている（概算でも可）
- [ ] dashboard/active-projects.md に登録されている

### Gate 2: 調査 → 企画
**判定者**: リサーチ + CEO
- [ ] 技術的な実現可能性が確認されている
- [ ] 主要な技術リスクが特定されている
- [ ] 推奨技術スタックが提示されている
- [ ] 競合分析が完了している（該当する場合）
- [ ] リサーチレポートが `reports/research-report.md` に保存されている

### Gate 3: 企画 → 承認
**判定者**: PM + マーケティング + CEO
- [ ] 提案書が完成している（`proposal.md`）
- [ ] 見積もり（工数・費用）が完了している
- [ ] スケジュール案が策定されている
- [ ] リスクと対策が整理されている
- [ ] 松竹梅プラン（3段階提案）が用意されている

### Gate 4: 承認 → 設計
**判定者**: CEO
- [ ] CEOのGo判定が下されている
- [ ] 選択されたプラン（松竹梅のどれか）が確定している
- [ ] クライアントの合意が得られている
- [ ] 正式なスケジュールが確定している
- [ ] decisions.md に承認記録がある

### Gate 5: 設計 → 開発
**判定者**: 開発 + レビュー
- [ ] 技術仕様書（`spec.md`）が完成している
- [ ] DB設計が完了している
- [ ] API設計が完了している
- [ ] 画面設計/ワイヤーフレームが完了している
- [ ] レビュー部門の設計レビューを通過している
- [ ] タスク一覧（`tasks.md`）が完成している

### Gate 6: 開発 → レビュー
**判定者**: 開発 + PM
- [ ] 全タスクの実装が完了している
- [ ] ユニットテストが通過している
- [ ] 主要機能の動作確認が完了している
- [ ] 技術ドキュメントが更新されている
- [ ] 既知の不具合が文書化されている

### Gate 7: レビュー → 納品
**判定者**: レビュー + CEO
- [ ] コードレビューでCriticalの指摘がゼロ
- [ ] セキュリティチェックを通過
- [ ] アクセシビリティチェックを通過
- [ ] パフォーマンス基準を満たしている
- [ ] E2Eテストが通過している
- [ ] 納品ドキュメントが完成している
- [ ] クライアントへの引き渡し手順が確認されている

## 品質ゲート不通過時の対応
1. 不通過の理由を明確にする
2. 対策を策定する
3. 対策を実施する
4. 再度品質ゲートの判定を受ける
5. 必要に応じてCEOにエスカレーションする
