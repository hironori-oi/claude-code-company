# プロジェクトライフサイクル

## 全体フロー

### 外部クライアント案件
```
受付 → 調査 → 企画 → 承認 → 設計 → 開発 → レビュー → 納品 → 運用保守
```

### 自社案件（簡易フロー）
```
内部起案 → 設計 → 開発 → レビュー → 納品 → 運用保守
```
※ 自社案件では Gate 1〜4 を「簡易Gate: 内部起案」に統合可能（詳細は `quality-gates.md` 参照）

## 各Phaseの詳細

### Phase 1: 受付
**主担当**: 秘書
**支援**: CEO
**期間目安**: 1日

#### やること
1. クライアントからの問い合わせ・依頼を受付
2. 案件ID（PRJ-XXX）を採番
3. プロジェクトディレクトリを作成
4. brief.md を作成（テンプレートに基づく）
5. dashboard を更新

#### 成果物
- `projects/{案件ID}/brief.md`
- `dashboard/active-projects.md` の更新

#### 次Phase移行条件
→ Gate 1（quality-gates.md参照）

---

### Phase 2: 調査
**主担当**: リサーチ
**支援**: 開発、マーケティング
**期間目安**: 2-3日

#### やること
1. 技術的実現可能性の調査
2. 推奨技術スタックの検討
3. 競合サービスの分析
4. 技術リスクの特定
5. 市場調査（必要に応じて）

#### 成果物
- `projects/{案件ID}/reports/research-report.md`
- `projects/{案件ID}/reports/marketing-report.md`（該当時）

#### 次Phase移行条件
→ Gate 2

---

### Phase 3: 企画
**主担当**: PM + マーケティング
**支援**: 開発（見積もり）、リサーチ（データ提供）
**期間目安**: 2-3日

#### やること
1. 提案書の作成（松竹梅プラン）
2. 工数見積もり（開発部門と連携）
3. スケジュール案の策定
4. リスクと対策の整理
5. 見積もり金額の算出

#### 成果物
- `projects/{案件ID}/proposal.md`
- `projects/{案件ID}/tasks.md`（初版）
- `projects/{案件ID}/risks.md`

#### 次Phase移行条件
→ Gate 3

---

### Phase 4: 承認
**主担当**: CEO
**支援**: 全部署（質問対応）
**期間目安**: 1日

#### やること
1. 提案書の最終レビュー
2. 見積もりの妥当性確認
3. Go/NoGo判定
4. クライアントへの提案（オーナーが実施）
5. クライアントの合意確認

#### 成果物
- `projects/{案件ID}/decisions.md` に承認記録

#### 次Phase移行条件
→ Gate 4

---

### Phase 5: 設計
**主担当**: 開発
**支援**: リサーチ、レビュー（設計レビュー）
**期間目安**: 3-5日

#### やること
0. **`projects/{案件ID}/app/` ディレクトリの作成** (アプリ実体はここに格納すること。詳細は `project-setup-checklist.md` の最優先ルール参照)
1. システムアーキテクチャ設計
2. データベース設計
3. API設計
4. 画面設計・UI/UXデザイン
5. 技術仕様書の作成
6. タスクの詳細分解
7. 設計レビュー（レビュー部門）

#### 成果物
- `projects/{案件ID}/spec.md`
- `projects/{案件ID}/tasks.md`（詳細版）

#### 次Phase移行条件
→ Gate 5

---

### Phase 6: 開発
**主担当**: 開発
**支援**: PM（進捗管理）、リサーチ（技術支援）
**期間目安**: 案件による

#### やること
1. 実装（フロントエンド、バックエンド、DB）
2. ユニットテストの作成・実行
3. 統合テスト
4. 技術ドキュメントの整備
5. 進捗報告（PM部門へ）

#### 成果物
- ソースコード
- テストコード
- `projects/{案件ID}/reports/dev-report.md`
- `projects/{案件ID}/progress.md` の更新

#### 次Phase移行条件
→ Gate 6

---

### Phase 7: レビュー
**主担当**: レビュー
**支援**: 開発（修正対応）
**期間目安**: 2-3日

#### やること
1. コードレビュー
2. セキュリティチェック（OWASP Top 10）
3. アクセシビリティチェック（WCAG 2.1 AA）
4. パフォーマンス検証
5. E2Eテスト
6. 指摘事項の修正確認

#### 成果物
- `projects/{案件ID}/reports/review-report.md`

#### 次Phase移行条件
→ Gate 7

---

### Phase 8: 納品
**主担当**: PM + 秘書
**支援**: 開発（デプロイ）、Web運営（ポートフォリオ）
**期間目安**: 1-2日

#### やること
1. 本番環境へのデプロイ
2. 納品ドキュメントの作成
3. クライアントへの引き渡し
4. 案件ステータスを「完了」に更新
5. ポートフォリオ掲載の検討（Web運営部門）
6. **KPT振り返りの実施（必須）** — Gate 7/9 通過後72時間以内に実施すること

#### 成果物
- 本番環境
- 納品ドキュメント
- `projects/{案件ID}/handover.md`
- **`organization/knowledge/{案件ID}-lessons-learned.md`（必須）**

#### KPT振り返りルール
- Gate 7（Web案件）または Gate 9（モバイル案件）通過後、72時間以内に実施する
- フォーマットは `organization/knowledge/prj-002-lessons-learned.md` を参照
- Keep / Problem / Try の3観点 + 横展開可能な技術パターンを記載
- 過去案件のKPTと重複しない新規知見を中心に記載する
- 新規案件のリサーチフェーズ（Gate 2）で、類似案件のKPTを必ず参照する

---

### Phase 9: 運用保守（該当する場合）
**主担当**: 開発
**支援**: PM（契約管理）
**期間目安**: 継続

#### やること
1. 監視・アラート対応
2. バグ修正
3. 機能改善リクエスト対応
4. パフォーマンス最適化
5. セキュリティアップデート

## 複数案件の並行管理
- 各案件は独立したPhaseで管理する
- リソース競合が発生した場合はCEOが優先順位を判断する
- 最大同時稼働案件数は状況に応じてCEOが調整する
