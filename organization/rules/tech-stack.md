# 技術スタック方針

## 基本方針
スピード・コスパ・柔軟性を重視し、以下を標準技術スタックとする。
案件要件に応じて柔軟に調整するが、特別な理由がない限りこの構成をベースとする。

## 標準技術スタック

## 技術選定ルール

### 新技術の導入基準
1. 標準スタックで対応できない場合のみ検討する
2. リサーチ部門が調査し、メリット/デメリットを比較する
3. CEOが最終判断する
4. 採用した場合は本ドキュメントを更新する

### 禁止事項
- jQuery の使用（レガシーコード除く）
- CSS-in-JS（styled-components, emotion 等）の新規採用
- 未メンテナンスのライブラリの採用（最終更新1年以上前）
- `any` 型の安易な使用（TypeScript の型安全性を維持）

### DB選定の判断基準
| 条件 | 選択 |
|------|------|
| 有料案件でリアルタイム機能が必要 | Supabase |
| 有料案件で認証+DB+ストレージを統合したい | Supabase |
| 無料枠でコストを抑えたい | Neon |
| DB のみ必要（認証は別手段） | Neon |

### ストレージ選定の判断基準
| 条件 | 選択 |
|------|------|
| Supabase を使っている案件 | Supabase Storage |
| Neon + Vercel の構成 | Vercel Blob |
| 大容量ファイル（5TB以下） | Vercel Blob |

## デザインシステム

### shadcn/ui 運用ルール
- `npx shadcn@latest init` でプロジェクト初期化
- 必要なコンポーネントを `npx shadcn@latest add {component}` で追加
- コンポーネントはプロジェクトのソースコードとして管理（カスタマイズ可能）
- `cn()` ユーティリティ（clsx + tailwind-merge）を使用
- カラーはCSS変数ベースのテーマを使用

### ダークモード/ライトモード
- 全案件でダーク/ライトモード切り替えをサポートする
- `next-themes` を使用してテーマ切り替えを実装
- `className="dark"` を `<html>` に適用する方式
- デフォルトは `system`（OS設定に追従）

### デザイン原則
- AI感が出すぎないクリーンなデザインを心がける
- ミニマルだが温かみのあるデザイン
- zinc/neutral 系のニュートラルカラーをベースに、アクセントカラーは1色
- タイポグラフィとスペーシングで階層を表現する
- モバイルファーストのレスポンシブデザイン
- アニメーションは控えめに（`framer-motion` は必要な場合のみ）
