# KPI定義

最終更新: 2026-05-05（PRJ-019 Round 19 で §E PRJ-019 専用 KPI 追加）
策定: PM部門 / §E は Web-Ops-F (Round 19 第 2 波)

---

## 方針

個人組織（1名運営）の規模に適したKPI設計とする。
- 計測負担を最小化するため、手動KPIは月次5分以内で記録できるものに絞る
- 最初は「必須KPI」のみ運用し、運用に慣れたら「拡張KPI」を追加する
- 定性的な実感と定量的な数値を組み合わせて判断する

## A. 納品品質メトリクス

### A-1. テストカバレッジ中央値（必須）

| 項目 | 内容 |
|------|------|
| 定義 | 完了案件のテストカバレッジ（%）の中央値 |
| 計測方法 | 手動：各案件の Gate 6 通過時のカバレッジ値を記録 |
| データソース | 各案件の `reports/` 内レビュー記録、またはCIカバレッジレポート |
| 目標値 | 75%以上（Level 2: 70% / Level 3: 80% の中間） |
| 警告閾値 | 65%未満 → 要注意フラグ |
| 更新頻度 | 案件完了ごと（月次集計） |
| 現在実績 | PRJ-003: 369テスト（実績あり）、PRJ-004: ビルドPASS |

### A-2. Gate不通過率（必須）

| 項目 | 内容 |
|------|------|
| 定義 | 各Gateで「初回不通過（差し戻し）」となった回数 ÷ Gate判定総数 |
| 計測方法 | 手動：Gate判定時に `decisions.md` に結果を記録し、月次集計 |
| データソース | 各案件の `decisions.md` |
| 目標値 | Gate不通過率 20%以下（5件に1件以下） |
| 警告閾値 | 40%以上 → 設計・開発プロセスの見直しが必要 |
| 更新頻度 | 月次 |
| 備考 | Gate別の傾向分析も実施（Gate 6・7 の不通過が多い場合は特に注意） |

### A-3. レビューCRITICAL検出数（必須）

| 項目 | 内容 |
|------|------|
| 定義 | コードレビューでCRITICALレベルの指摘が検出された件数（案件別・累計） |
| 計測方法 | 手動：各案件のレビューレポートから抽出 |
| データソース | `projects/{案件ID}/reports/` 内のレビューレポート |
| 目標値 | 案件あたりCRITICAL検出 0件（Gate 7 通過条件と同一） |
| 警告閾値 | 1案件あたり3件以上 → 設計・実装プロセスの根本見直し |
| 更新頻度 | 案件完了ごと |
| 備考 | CRITICAL件数の傾向（セキュリティ/ロジック/など分類）も記録する |

### A-4. 本番障害件数（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | 納品・デプロイ後30日間に発生した本番障害（ユーザー影響あり）の件数 |
| 計測方法 | 手動：障害発生時に記録（ゼロが続く場合は記録不要） |
| データソース | 案件別の障害ログ（Sentry等、または手動記録） |
| 目標値 | 0件 |
| 警告閾値 | 1件以上 → 即時根本原因分析（RCA）を実施 |
| 更新頻度 | 随時（障害発生時）+ 四半期サマリー |
| 備考 | 現状は自社プロダクト中心のため影響範囲は限定的。外部クライアント案件が増えたら必須化する |

### A-5. Core Web Vitals合格率（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | Webアプリ案件でCore Web Vitals基準（LCP<2.5s, CLS<0.1, INP<200ms）を全項目通過した案件の割合 |
| 計測方法 | 手動：Gate 7 通過時に Lighthouse / PageSpeed Insights の結果を記録 |
| データソース | 各案件の Gate 7 判定記録 |
| 目標値 | 100%（全Webアプリ案件で通過） |
| 警告閾値 | 80%未満（5件中1件以上が不合格） |
| 更新頻度 | 案件完了ごと |
| 備考 | モバイルアプリ（Expo/RN）は対象外 |

## B. 開発効率メトリクス

### B-1. リードタイム（必須）

| 項目 | 内容 |
|------|------|
| 定義 | Gate 1（または簡易Gate）通過日から Gate 7 通過日までの実日数 |
| 計測方法 | 手動：各案件の `decisions.md` から開始日・完了日を記録 |
| データソース | `projects/{案件ID}/decisions.md`、`dashboard/active-projects.md` |
| 目標値 | 案件規模別に設定（小規模LP: 2週間以内 / 標準Webアプリ: 6週間以内 / 大規模: 12週間以内） |
| 警告閾値 | 目標値の150%超過 → スコープ・見積もりの見直し |
| 更新頻度 | 案件完了ごと（四半期サマリー） |
| 備考 | 待機時間（クライアント確認待ち等）を除いた「実稼働日数」も別途記録すると改善分析に有効 |

### B-2. 案件あたりのタスク数・テスト数（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | 案件完了時のタスク総数（`tasks.md` 行数）とテスト総数 |
| 計測方法 | 手動：案件完了時に `tasks.md` とテストファイルから集計 |
| データソース | 各案件の `tasks.md`、CIテスト結果 |
| 目標値 | 参考値（規模感の把握が目的、目標値なし） |
| 警告閾値 | テスト数/タスク数の比率 < 2.0 → テスト不足の可能性 |
| 更新頻度 | 案件完了ごと |
| 備考 | PRJ-003（83タスク/369テスト = 4.4倍）を基準値として蓄積する |

### B-3. 簡易Gate→設計開始までの日数（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | 自社案件における簡易Gate通過日から Gate 5（設計完了）日までの日数 |
| 計測方法 | 手動：`decisions.md` から抽出 |
| データソース | `projects/{案件ID}/decisions.md` |
| 目標値 | 5営業日以内 |
| 警告閾値 | 10営業日超 → 起案〜設計フローのボトルネック調査 |
| 更新頻度 | 案件完了ごと |

### B-4. KPT実施率（必須）

| 項目 | 内容 |
|------|------|
| 定義 | 完了案件のうちKPTレポートを作成した案件の割合 |
| 計測方法 | 手動：`organization/knowledge/` または案件フォルダのKPTファイル有無を確認 |
| データソース | `organization/knowledge/`、`projects/{案件ID}/` |
| 目標値 | 100%（全完了案件でKPT実施） |
| 警告閾値 | 80%未満（5件中1件以上が未実施） |
| 更新頻度 | 月次 |
| 備考 | KPT未実施は「案件完了」と見なさない運用を推奨 |

## C. ビジネスメトリクス

### C-1. 案件完了数（必須）

| 項目 | 内容 |
|------|------|
| 定義 | Gate 7 通過（または App Store リリース）を迎えた案件数 |
| 計測方法 | 手動：`dashboard/active-projects.md` の「最近完了した案件」テーブルから集計 |
| データソース | `dashboard/active-projects.md` |
| 目標値 | 月1件以上（四半期3件以上） |
| 警告閾値 | 四半期0件 → パイプライン・スコープ設定の見直し |
| 更新頻度 | 月次 |

### C-2. 見積もり精度（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | （実工数 - 見積もり工数）÷ 見積もり工数 × 100（%） |
| 計測方法 | 手動：案件完了時に見積もり工数と実際のタスク数・日数を比較 |
| データソース | `projects/{案件ID}/proposal.md`（見積もり）、`decisions.md`（実績） |
| 目標値 | ±30%以内 |
| 警告閾値 | +50%超（大幅超過）または -50%未満（大幅過剰見積もり） |
| 更新頻度 | 案件完了ごと（四半期サマリー） |
| 備考 | 自社案件（見積もりなし）は計測対象外。外部クライアント案件から運用開始 |

### C-3. ポートフォリオ掲載率（必須）

| 項目 | 内容 |
|------|------|
| 定義 | 完了案件のうち自社HP（COMPANY-WEBSITE）にポートフォリオとして掲載された案件の割合 |
| 計測方法 | 手動：`dashboard/active-projects.md` のポートフォリオ列を確認 |
| データソース | `dashboard/active-projects.md`、自社HP |
| 目標値 | 80%以上（クライアント非公開案件は対象外） |
| 警告閾値 | 50%未満 → web-ops部門へのタスク依頼が滞っている |
| 更新頻度 | 月次 |

### C-4. クライアント満足度（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | 外部クライアント案件完了後のフィードバックスコア（5段階評価） |
| 計測方法 | 手動：納品後のメールフォローアップで確認 |
| データソース | クライアントからのフィードバックメール |
| 目標値 | 平均4.0以上 |
| 警告閾値 | 3.0未満（または否定的フィードバックの発生） |
| 更新頻度 | 案件完了ごと |
| 備考 | 現在は自社プロダクト中心のため計測対象案件なし。外部案件が発生したら即時運用開始 |

## D. 技術負債メトリクス

### D-1. npm audit 脆弱性数（必須）

| 項目 | 内容 |
|------|------|
| 定義 | `npm audit --audit-level=high` で検出された HIGH以上の脆弱性件数（案件別） |
| 計測方法 | 自動（Gate 7 チェックリストの一部として既存）＋月次スナップショット |
| データソース | `npm audit` 出力、GitHub Dependabot アラート |
| 目標値 | HIGH以上 = 0件（Gate 7 通過条件と同一） |
| 警告閾値 | HIGH以上が1件でも検出された場合 → 即時対応 |
| 更新頻度 | 月次（Dependabotアラートは随時） |

### D-2. TypeScript any型使用数（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | プロジェクト内のソースコード（`src/`等）における `any` 型の使用箇所数 |
| 計測方法 | 手動（任意）：`grep -r ": any" src/ \| wc -l` で計測 |
| データソース | ソースコード |
| 目標値 | 案件あたり 0件（TypeScript strict mode 適用時） |
| 警告閾値 | 10件以上 → 型安全性の再確認 |
| 更新頻度 | 四半期（Gate 7 チェック時に確認） |
| 備考 | 外部ライブラリの型定義由来は除外して計測する |

### D-3. TODO/FIXMEコメント数（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | ソースコード内の `TODO:` / `FIXME:` / `HACK:` コメントの総数 |
| 計測方法 | 手動（任意）：`grep -rn "TODO\|FIXME\|HACK" src/` で計測 |
| データソース | ソースコード |
| 目標値 | 納品時 0件 |
| 警告閾値 | 5件以上 → 技術負債化のリスクあり |
| 更新頻度 | 四半期（Gate 7 チェック時に確認） |

### D-4. 依存パッケージのメジャーバージョン遅延数（拡張）

| 項目 | 内容 |
|------|------|
| 定義 | `package.json` の主要依存パッケージ（Next.js, React, Expo等）で最新メジャーバージョンから2世代以上遅れているものの数 |
| 計測方法 | 手動：四半期ごとに主要パッケージの最新版を確認 |
| データソース | `package.json`、npmレジストリ（npmjs.com） |
| 目標値 | 0件（常に最新または1世代以内） |
| 警告閾値 | 2件以上 → アップグレード計画を策定 |
| 更新頻度 | 四半期 |

## E. PRJ-019 Round 19 専用 KPI（公開直前 trajectory 計測）

### E-1. 17 日 path W3 完成度（Round 19 必須）

| 項目 | 内容 |
|------|------|
| 定義 | 公開当日 (2026-06-19) までの 17 日 path（Round 16 起点〜W3 着地）における W3 完成度（％） |
| 計測方法 | 各 Round 完遂時に `dashboard/active-projects.md` の進捗％ を更新、W3 = 95% を完成判定 |
| データソース | `dashboard/active-projects.md`、Round 別 ceo 着地レポート |
| 目標値 | 6/19 公開時点で 95% 以上 |
| 警告閾値 | 6/15 (D-4) 時点で 90% 未満 → リスク再評価 |
| 更新頻度 | Round ごと（現状 1〜2 day cadence） |
| 現在実績 | Round 18 着地時 92%（Round 17→18 で +2pt） |

### E-2. Heartbeat 500k SLO 適合率（Round 19 必須）

| 項目 | 内容 |
|------|------|
| 定義 | cron heartbeat（5 本）の 30 日 rolling window での 500k token / day 上限への適合率 |
| 計測方法 | Vercel Cron logs から token consumption を集計、SLO = 500k/day を超過しない day 数 ÷ 観測 day 数 |
| データソース | Vercel Cron Dashboard、`runbooks/cron-fallback-switch.md` 由来 alert |
| 目標値 | 公開〜30 日で 100%（1 day も超過しない） |
| 警告閾値 | 95% 未満（30 日中 2 day 以上超過）→ fallback switch 発動 |
| 更新頻度 | 日次自動集計、月次 manual 確認 |
| 備考 | DEC-019-062 で確定した cron 5 本（CRON_SECRET 経由）の運用 SLO。alert は Slack `#prj-019-alerts` |

### E-3. DEC 採択 trajectory（Round 19 拡張）
