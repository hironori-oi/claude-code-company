# PRJ-002 スキャンPDF帳票 位置ずれ補正機能 調査報告書

**作成日**: 2026-04-02
**作成部門**: リサーチ部門 / 開発部門（CEO統括）
**対象**: PDF帳票OCR/VLMデータ自動抽出システム

---

## 1. 問題の定義

### 現象
スキャナで取り込んだPDF帳票は、同一フォーマットであっても以下のばらつきが発生する:

| ずれの種類 | 実測範囲（サンプル3件から） | 影響度 |
|-----------|-------------------------|-------|
| 水平方向の位置ずれ | -12mm ~ +12mm | **高** |
| 垂直方向の位置ずれ | -3mm ~ +3mm | 中 |
| 回転（スキュー） | -2度 ~ +4度 | **高** |
| スケール（拡大縮小） | 100% ~ 110% | 中 |
| 影・ノイズ | 軽微～中程度 | 低 |

### 現行システムの限界
現行の抽出パイプラインは以下のフローで動作する:

```
テンプレート定義（% 座標） → cropPdfRegion（固定scale=3.0）→ VLM/OCR
```

**% 座標は「PDFページ全体に対する相対位置」** であり、スキャンずれが発生すると帳票の内容がページ内で移動するため、指定した領域と実際のフィールド位置が一致しなくなる。

例: フリガナ欄を `x:15%, y:12%` で指定していた場合、帳票が右に12mmずれると、クロップ画像にフリガナ欄の左端が欠ける。

---

## 2. 解決策の一覧と評価

### 方式A: アンカーポイント自動検出による座標補正

**概要**: 帳票上の固定的な特徴点（罫線の交点、ロゴ、QRコード等）を「アンカー」として自動検出し、テンプレート基準画像との差分から変換行列を算出。全フィールド座標をリアルタイムに補正する。

**技術的実装**:
1. テンプレート作成時に基準PDFから特徴点を抽出・保存
2. 抽出実行時にスキャンPDFから同じ特徴点を検出
3. アフィン変換行列（平行移動 + 回転 + スケール）を計算
4. 全フィールドの % 座標を変換後の座標に補正してからクロップ

**特徴点の候補**:
- 帳票の外枠（黒い罫線の四隅）
- セクション区切りの太線
- 固定テキスト（タイトル文字列の位置）
- QRコード / バーコード（存在する場合）

**評価**:
| 項目 | 評価 |
|------|------|
| 補正精度 | 非常に高い（サブピクセル精度可能） |
| 対応できるずれ | 平行移動、回転、スケール、台形歪み |
| 実装コスト | 高（画像処理ライブラリ必要） |
| 処理速度への影響 | 中（1ページあたり100-500ms追加） |
| ユーザー操作 | テンプレート作成時にアンカー指定が必要 |

**必要技術**: Canvas API でのエッジ検出、またはサーバーサイドでの OpenCV / sharp 活用

---

### 方式B: 罫線ベースの自動位置合わせ（帳票枠検出）

**概要**: 帳票の最外枠（黒い太線の矩形）をCanvasベースの画像処理で自動検出し、枠の位置・サイズ・回転角からページ全体の変換を算出。座標を枠基準に再計算する。

**技術的実装**:
1. PDFを画像化（現行のcropPdfRegionと同じscale）
2. グレースケール → 二値化 → 輪郭検出
3. 最大面積の矩形輪郭を帳票枠として特定
4. 枠の四隅座標から回転角・スケール・オフセットを計算
5. 全フィールド座標を枠基準の相対座標に変換

**評価**:
| 項目 | 評価 |
|------|------|
| 補正精度 | 高い（枠がはっきりしている帳票に有効） |
| 対応できるずれ | 平行移動、回転、スケール |
| 実装コスト | 中（Canvas APIのみで実装可能） |
| 処理速度への影響 | 低（1ページあたり50-200ms追加） |
| ユーザー操作 | 不要（自動検出） |
| 前提条件 | **帳票に明確な外枠が存在すること** |

**利点**: 今回のアンケート帳票は明確な黒枠を持つため、高い検出精度が期待できる。
**制約**: 枠なし帳票（自由フォーマット）には対応できない。

---

### 方式C: VLMによるインテリジェント領域拡張

**概要**: クロップ領域を意図的に拡大（マージン追加）してVLMに送信し、VLMの文脈理解力で正しいフィールドを抽出させる。

**技術的実装**:
1. 各フィールドのクロップ領域に上下左右にマージン（例: 各方向+20%）を追加
2. 拡大されたクロップ画像をVLMに送信
3. プロンプトで「指定フィールドの値のみ抽出」と明示的に指示
4. VLMが周囲のラベルやレイアウトから正しい値を判断

**評価**:
| 項目 | 評価 |
|------|------|
| 補正精度 | 中～高（VLMの判断力に依存） |
| 対応できるずれ | 平行移動（マージン範囲内） |
| 実装コスト | **非常に低い**（既存コード微修正のみ） |
| 処理速度への影響 | 低（画像サイズ増加のみ） |
| ユーザー操作 | 不要 |
| API費用 | 微増（画像トークン増加） |
| 前提条件 | VLMモードでのみ有効 |

**利点**: 最小の実装コストで即座に効果を発揮。
**制約**: 回転・スケールのずれには対応不可。マージンが大きすぎると隣接フィールドの値を誤抽出するリスク。

---

### 方式D: スキュー（傾き）自動補正

**概要**: スキャン画像の傾きを検出し、クロップ前に画像全体を回転補正する前処理。

**技術的実装**:
1. PDFを画像化
2. ハフ変換またはプロジェクション解析で主要な直線を検出
3. 直線群の平均角度からスキュー角を算出
4. Canvas上で逆回転を適用
5. 補正済み画像に対して通常のクロップ処理を実行

**評価**:
| 項目 | 評価 |
|------|------|
| 補正精度 | 高い（回転に特化） |
| 対応できるずれ | **回転のみ** |
| 実装コスト | 中 |
| 処理速度への影響 | 低～中（1ページあたり50-150ms追加） |
| ユーザー操作 | 不要 |

**利点**: 罫線が多い帳票では高精度にスキュー角を検出可能。
**制約**: 位置ずれ・スケール変動は別途対処が必要。

---

### 方式E: テンプレートマッチング（画像相関）

**概要**: テンプレート基準画像とスキャン画像の相関係数を計算し、最もマッチする位置を特定する。

**技術的実装**:
1. テンプレート作成時に基準画像を保存
2. 抽出時に正規化相互相関（NCC）でマッチング
3. マッチ位置から平行移動量を算出
4. 座標を補正

**評価**:
| 項目 | 評価 |
|------|------|
| 補正精度 | 高い |
| 対応できるずれ | 平行移動（回転は事前補正が必要） |
| 実装コスト | 高（画像処理の実装が複雑） |
| 処理速度への影響 | 高（計算コスト大） |
| ユーザー操作 | 不要 |

**制約**: ブラウザ上でのリアルタイム実行は計算量的に厳しい。サーバーサイド処理が必要になる可能性が高い。

---

### 方式F: 複合アプローチ（推奨）

**概要**: 上記を段階的に組み合わせ、コスト対効果を最適化する。

**Phase 1（即時対応）: 方式C - VLMマージン拡張**
- 実装コスト: 1日以内
- 効果: 軽微なずれ（±10mm程度）に対応
- VLMモード使用時に即座に効果

**Phase 2（短期対応）: 方式D - スキュー自動補正**
- 実装コスト: 3-5日
- 効果: 回転ずれ（±5度）を解消
- Canvas APIベースでブラウザ上で完結

**Phase 3（中期対応）: 方式B - 罫線ベース自動位置合わせ**
- 実装コスト: 1-2週間
- 効果: 平行移動 + スケール補正
- 枠付き帳票で高精度な位置合わせを実現

**Phase 4（長期対応）: 方式A - アンカーポイント**
- 実装コスト: 2-3週間
- 効果: あらゆるずれに対応可能な汎用解
- 枠なし帳票にも対応

## 3. 技術的実現性の詳細分析

### ブラウザ上で実行可能な画像処理

現行システムはクライアントサイドで `OffscreenCanvas` + `pdf.js` を使用してPDFを画像化している。以下の処理はブラウザ上で実装可能:

| 処理 | Canvas API | 追加ライブラリ | 実現性 |
|------|----------|-------------|-------|
| グレースケール変換 | `getImageData` | 不要 | 容易 |
| 二値化（Otsu法） | `getImageData` | 不要 | 容易（OCRエンジンに実装済み） |
| エッジ検出（Sobel） | `getImageData` | 不要 | 中程度 |
| 輪郭検出 | `getImageData` | 不要 | 中程度 |
| 矩形検出 | 輪郭データから | 不要 | 中程度 |
| アフィン変換 | `setTransform` | 不要 | 容易 |
| ハフ変換（直線検出） | `getImageData` | 不要 | やや複雑 |
| テンプレートマッチング | `getImageData` | 不要 | 複雑・重い |

### サーバーサイドが必要になるケース
- 大量のバッチ処理（100ファイル以上）で前処理のパフォーマンスが問題になる場合
- より高度な画像処理（射影変換、高精度なテンプレートマッチング）が必要な場合

## 4. 競合製品の対応状況

| 製品 | ずれ補正方式 | 備考 |
|------|------------|------|
| AWS Textract | 自動（内部的にスキュー補正 + 領域検出） | クラウドAPI、高価格 |
| Google Document AI | 自動（ML ベースのレイアウト解析） | クラウドAPI |
| ABBYY FineReader | アンカーポイント + 自動スキュー補正 | デスクトップ/クラウド |
| AI-OCR（国内SaaS） | テンプレートマッチング方式が主流 | 枠検出ベースが多い |

**差別化ポイント**: 当社はVLMを活用した「インテリジェントな領域認識」ができるため、固定枠依存のテンプレートマッチングよりも柔軟な対応が可能。これは競合にない強みとなる。

## 5. 推奨方針（CEO判断）

**複合アプローチ（方式F）** を採用し、Phase 1から段階的に実装する。

### 即時着手: Phase 1（VLMマージン拡張）
- **理由**: 最小コストで最大効果。現在のサンプル帳票のずれ範囲（±12mm）の大半をカバー可能
- **リスク**: 隣接フィールドの誤抽出（マージン量のチューニングで対処）

### 次期対応: Phase 2（スキュー補正）+ Phase 3（枠検出）
- **理由**: Phase 1では対応できない回転ずれと大きな位置ずれに対応
- **判断時期**: Phase 1のユーザーフィードバックを踏まえて判断
