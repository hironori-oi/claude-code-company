## 1. swapSlots / moveSlot 実装概要

`lib/stores/workspace-layout.ts` に 2 つの新 action を追加。いずれも current
project x session の `SessionLayout.slots` を不変更新する純粋な state 操作。

```ts
swapSlots: (indexA, indexB) => set((s) => {
  if (indexA === indexB) return s;
  if (indexA < 0 || indexA >= MAX_WORKSPACE_SLOTS) return s;
  if (indexB < 0 || indexB >= MAX_WORKSPACE_SLOTS) return s;
  const { projectKey, sessionKey } = getCurrentLayoutKeys();
  const cur = readLayout(s.layouts, projectKey, sessionKey);
  if (!cur) return s;
  const next = cur.slots.slice();
  const tmp = next[indexA];
  next[indexA] = next[indexB] ?? null;
  next[indexB] = tmp ?? null;
  return { layouts: writeLayout(s.layouts, projectKey, sessionKey, { ...cur, slots: next }) };
})
```

`moveSlot` は同様に `next[toIndex] = next[fromIndex]; next[fromIndex] = null;`。
DEC-055 の project x session キー設計、DEC-058 の cascade ルール、PM-980 の
「同 chip 1 slot」制約と整合し、既存 `setSlot` 経路には触れていない。

## 2. TrayBar chip の sourceSlotIndex 逆引き

`TrayChips()` 内で `placedRefs` (Set<string>) と並行して `placedSlotIndex`
(Map<string, number>) を `useMemo([slots, layout])` で構築。visible slot のみを
走査し `${kind}:${refId}` → `slotIndex` で逆引きを保持する。

`FixedChip` / `EditorChip` に `sourceSlotIndex: number | null` を prop で渡し、
`useDraggable.data` を以下に拡張:

```ts
data: { type: "tray-chip", kind, refId, label, sourceSlotIndex }
```

未配置の chip は `sourceSlotIndex = null` (= 既存経路)、配置中の chip は
slot index を持つ。これで TrayBar 経由の drag でも「配置中のものが移動する」
意味付けが drop 側で判別できる。

## 3. SlotContainer drag handle UI

`SlotContainer.tsx` に `useDraggable({ id: 'slot-content-N', data: { type: 'slot-content', sourceSlotIndex, kind, refId, label } })`
を追加。`disabled: !content` で空 slot ではドラッグ不可。

`SlotHeader` に `lucide-react/GripVertical` のボタンを追加 (h-4 w-4、薄表示、
hover で foreground 色)。`aria-label="Slot {label} の中身をドラッグして移動・入替"`。
`SlotContainer` 全体に `isDraggingThisSlot && "opacity-50"` を適用、ドラッグ中
は半透明表示で視覚フィードバック。droppable 側も既存 `border-primary/60 bg-primary/5`
が dnd-kit の `isOver` で点灯する。

## 4. handleDragEnd の 4 経路分岐

`WorkspaceView.tsx` の `handleDragEnd` を拡張。

1. `over.data.slotIndex` を取得し、`VISIBLE_SLOTS[layout]` に含まれない場合は no-op
   (不可視 slot index への drop 防御)。
2. `active.data.sourceSlotIndex` が number → 既配置経路:
   - self-drop (source === target) は no-op
   - source が不可視 slot のときも no-op
   - target が空 → `moveSlot(source, target)`
   - target が占有 → `swapSlots(source, target)`
3. `sourceSlotIndex` が null → 未配置 chip 経路:
   - `refId` あり → `setSlot(target, { kind, refId })` (空 slot で配置 / 占有 slot で置換)
   - `refId` なし & kind === "terminal" → `createTerminal()` lazy spawn → setSlot
   - `refId` なし & kind === "preview" → `addInstance()` lazy spawn → setSlot

`handleDragStart` も `type: "slot-content"` の場合は data から kind/label を
取り出して DragOverlay に表示する。

## 注意事項とフォローアップ

- v1.31.0 では D&D feedback として toast.info は省略（オーナー指示の「過剰なら省略」を
  尊重）。実機操作で十分視覚的に分かるため。
- swap 時、preview / terminal / editor の **instance は state 内で
  そのまま入替**。React 側で SlotContainer 内の `<PreviewPane />` / `<TerminalPane />` /
  `<FileViewer />` は `key` prop を持たないため、kind が同じなら React は再 mount
  しないが、kind が変わる swap (例: chat ↔ preview) では SlotContentRenderer 内で
  別 component に切替えるため必然的に re-mount される。pty は backend 側で生存し、
  iframe URL は instance id ベースで保持されるので、ユーザー視点では「同じ内容が
  位置だけ変わった」体験になる。
- 不可視 slot への drop は `VISIBLE_SLOTS[layout]` で防御。layout を 4 → 2h に
  狭めた直後の stale な slot 2/3 は Drop target として無効化される。
- 既存の lazy spawn (terminal / preview の `null refId` 経路) は **未配置 chip
  drop の経路でのみ** 動作する。配置中 chip の drag は `sourceSlotIndex` を持つ
  ので lazy spawn 分岐に入らない。
