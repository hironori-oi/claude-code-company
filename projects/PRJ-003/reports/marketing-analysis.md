## 1. ターゲット市場分析

### 1.1 不定様式帳票処理のペインポイント

企業が日常的に受け取る帳票は、取引先ごとにフォーマットが異なる「不定様式」が大半を占める。定型帳票のOCR/AI-OCRが普及し始めた現在でも、不定様式帳票は依然として手作業に頼らざるを得ない領域である。

| ペインポイント | 詳細 | 深刻度 |
|--------------|------|--------|
| **テンプレート作成の手間** | 取引先ごとに帳票様式が異なり、テンプレートを都度定義する必要がある | 極めて高い |
| **テンプレート管理コスト** | 取引先数 x 帳票種類のテンプレートが必要、メンテナンスが負担 | 極めて高い |
| **新規取引先への対応遅延** | 新しい取引先の帳票が届いてからテンプレート作成までにリードタイムが発生 | 高い |
| **様式変更への追従** | 取引先が帳票フォーマットを変更した場合の再設定が必要 | 高い |
| **スケーラビリティの欠如** | 取引先が増えるほどテンプレート管理が破綻する | 高い |
| **属人化** | テンプレート定義やルール設定がベテラン担当者に依存 | 中-高 |
| **多様な帳票種への未対応** | 発注書、見積書、納品書、請求書など帳票種類ごとに対応が必要 | 中-高 |

### 1.2 ターゲット業種と業務シーン

#### Tier 1: 最優先ターゲット

**物流・運送業**
- **業務シーン**: 複数の荷主から届く配送依頼書、送り状、入出庫伝票の処理
- **ペイン**: 荷主ごとに伝票フォーマットが全く異なる。数十~数百の取引先に対しテンプレート定義は非現実的
- **抽出項目例**: 荷主名、届け先住所、品名、数量、重量、配送希望日
- **推定市場**: 国内運送事業者 約63,000社、うちDX意欲のある中小規模 約10,000社

**製造業（仕入れ・調達部門）**
- **業務シーン**: 多数のサプライヤーから届く見積書・納品書・請求書の突合処理
- **ペイン**: サプライヤーごとの帳票様式の違い、品番体系の差異
- **抽出項目例**: 得意先名、品目名、品番、数量、単価、納期、合計金額
- **推定市場**: 中小製造業 約180,000社、仕入れ帳票処理の自動化ニーズ約30,000社

**小売・卸売業**
- **業務シーン**: 仕入先からの納品書・請求書、テナントからの売上報告書
- **ペイン**: 仕入先が多いほど帳票バリエーションが膨大
- **抽出項目例**: 商品名、JANコード、数量、単価、消費税区分
- **推定市場**: 中小小売・卸売業 約120,000社、対象約20,000社

#### Tier 2: 中期ターゲット

**士業事務所（税理士・会計士）**
- **業務シーン**: 顧問先企業の多種多様な領収書・請求書の処理
- **ペイン**: 顧問先ごとに異なる取引先の帳票を大量処理。顧問先が増えるほどバリエーション爆発
- **推定市場**: 税理士事務所 約26,000、うちDX志向 約8,000

**建設業**
- **業務シーン**: 下請け業者からの出来高報告書、資材発注書、安全書類
- **ペイン**: 下請け構造が多層的で書類フォーマットが統一困難
- **推定市場**: 中小建設業 約470,000社、対象約30,000社

#### Tier 3: 長期ターゲット

- **医療機関**: 紹介状、検査報告書（個人情報保護のためローカルAI需要が高い）
- **不動産業**: 物件情報シート、重要事項説明書の項目抽出
- **自治体・公共機関**: 申請書、届出書の多様なフォーマット処理

### 1.4 PRJ-002（定型帳票向け）との市場棲み分け

| 観点 | PRJ-002（定型帳票） | PRJ-003（不定様式） |
|------|-------------------|-------------------|
| **ユースケース** | 自社の帳票フォーマットが固定、社内文書処理 | 取引先からの多様な帳票受領処理 |
| **テンプレート** | 必要（事前定義） | 不要（抽出項目を指定するだけ） |
| **精度優先度** | 極めて高い（固定レイアウトで高精度可能） | 高い（柔軟性とのトレードオフ） |
| **主要顧客** | 自社帳票を大量に処理する企業 | 多数の取引先から帳票を受け取る企業 |
| **導入の手軽さ** | テンプレート設定が必要 | 抽出項目を選ぶだけで即利用可能 |
| **拡張性** | 帳票追加ごとにテンプレート定義 | 新しい帳票が来てもそのまま処理 |

**重要な洞察**: PRJ-002とPRJ-003は競合ではなく補完関係にある。定型帳票（社内文書、自社発行帳票）はPRJ-002で高精度処理し、不定様式帳票（取引先からの受領文書）はPRJ-003で柔軟に処理する。一つのプラットフォームで両方提供できることが最大の差別化となる。

### 2.4 競合の参入障壁と差別化余地

PRJ-003にとっての実質的な競合は **Amazon Textract Queries** と **Google Document AI Custom Extractor** である。両者はコンセプトが似ているが、以下の差別化余地がある:

1. **日本語帳票への最適化**: 海外勢は日本語の縦書き、全角半角混在、日本固有のレイアウト（帳票の罫線構造等）への対応が弱い
2. **ローカルAI対応**: 海外クラウドサービスにはない、データを外部に出さない選択肢
3. **完成されたUI**: API Onlyではなく、非エンジニアが使えるWebアプリとして提供
4. **PRJ-002との統合**: 定型+不定型を一つのプラットフォームで提供できる唯一のソリューション

## 3. 価値提案・差別化戦略

### 3.1 コア・バリュー・プロポジション

> **多数の取引先から異なるフォーマットの帳票を受け取る中小企業**のために、
> **PRJ-003** は、
> **テンプレート定義なしでPDFから必要なデータを自動抽出するツール**です。
>
> 既存のAI-OCRサービスとは異なり、
> **「何を抽出したいか」を指定するだけで、VLMがレイアウトを自動理解し、
> どんなフォーマットの帳票からも同じ設定で情報を取り出せます。**

### 3.2 「テンプレート定義不要」の価値

テンプレート不要であることの具体的な顧客価値:

| 価値 | 定量的インパクト |
|------|----------------|
| **導入時間の大幅短縮** | テンプレート作成（1帳票あたり30分-1時間）が不要。即日利用開始可能 |
| **運用コストの削減** | テンプレート管理・メンテナンスの工数がゼロ |
| **新規取引先の即時対応** | 新しいフォーマットの帳票が届いても設定変更なしで処理可能 |
| **スケーラビリティ** | 取引先100社でも1,000社でも同じ設定で対応可能 |
| **属人化の排除** | テンプレート定義スキルが不要。誰でも使える |

**訴求メッセージ**: 「取引先が何社でも、設定は一つだけ」

### 3.3 中小企業向け価格設定の方向性

#### SaaSモデル（推奨）

| プラン | 月額（税抜） | 処理枚数/月 | 抽出項目数 | 主な対象 |
|--------|------------|------------|-----------|---------|
| **フリー** | 0円 | 30枚 | 5項目 | 導入検討・評価 |
| **スターター** | 4,980円 | 200枚 | 10項目 | 個人事業主・小規模事業者 |
| **ビジネス** | 14,800円 | 1,000枚 | 20項目 | 中小企業（推奨） |
| **プロ** | 39,800円 | 3,000枚 | 無制限 | 中堅企業・士業事務所 |
| **エンタープライズ** | 要相談 | 無制限 | 無制限 | 大量処理・カスタム要件 |

**価格設定の根拠**:
- PRJ-002より低い価格帯を設定（不定様式は精度面でのリスクがあるため、導入障壁を下げる）
- Textract Queries（1ページ$0.01-$0.02）と比較して、UI/UXの付加価値を価格に反映
- ローカルAI利用時のAPI費用ゼロを活用した低価格戦略
- フリープランの制限を厳しめにし、有料転換率を高める

**訴求**: 「定型も不定型も、一つのプラットフォームで。セット割で最大30%OFF」

### 3.4 セルフサービス vs エンタープライズ

| 観点 | セルフサービス（主力） | エンタープライズ |
|------|---------------------|----------------|
| **導入方法** | Webサインアップ → 即利用 | 営業対応 → PoC → 導入 |
| **ターゲット** | 中小企業（従業員100名以下） | 中堅～大企業 |
| **サポート** | ヘルプドキュメント + チャット | 専任担当 + オンボーディング |
| **カスタマイズ** | UI上でセルフ設定 | 個別カスタマイズ対応 |
| **優先度** | **Phase 1で注力** | Phase 3以降 |

### 3.5 PRJ-002との統合メリット（プラットフォーム戦略）

PRJ-002（定型帳票）とPRJ-003（不定様式）を統合したプラットフォームとして提供する戦略的メリット:

1. **顧客の業務カバレッジ拡大**: 一つのツールで帳票処理を完結できる
2. **クロスセル収益**: PRJ-002ユーザーへのPRJ-003アップセル、その逆も
3. **競合との決定的差別化**: 定型+不定型の両方を一つのプラットフォームで提供できる競合は存在しない
4. **データ統合**: 定型・不定型のデータを統一フォーマットで出力し、後続処理を一元化
5. **ロックイン効果**: 両方使うことで乗り換えコストが上昇し、チャーンレート低下

## 4. Go-to-Market戦略

### 4.1 初期ターゲットの優先順位

#### 最優先: 物流・運送業の仕入れ/配車担当

**選定理由**:
- 帳票のフォーマットバリエーションが最も多い業種の一つ
- テンプレート不要の価値が最も刺さる（荷主ごとに伝票フォーマットが異なる）
- 抽出項目が比較的シンプル（荷主名、届け先、品名、数量等）で初期精度を出しやすい
- DX遅延が顕著で、手作業への不満が大きい
- 業界のIT投資意欲が上昇中

**攻め方**:
- 「配送依頼書の自動読取」を初期ユースケースとして特化
- 物流業界メディア（物流ニッポン、LNEWS等）でのPR
- 物流DXセミナーへの登壇
- 運送会社3-5社でのPoC実施、成功事例をケーススタディ化

#### 次点: 製造業の仕入れ担当

**選定理由**:
- サプライヤーからの見積書・納品書の処理ニーズが大きい
- 品番・数量・単価の抽出は精度検証がしやすい
- 製造業のDX補助金を活用した導入促進が可能

#### 第三: 士業事務所（PRJ-002からのクロスセル）

**選定理由**:
- PRJ-002で既に接点がある顧客層
- 顧問先の多様な帳票処理が最大のペイン
- セキュリティ要件が高く、ローカルAI対応が刺さる

### 4.2 フリーミアムモデルの適用可能性

| 要素 | 設計方針 |
|------|---------|
| **無料枠の目的** | 製品価値の実感、導入障壁の排除、口コミ獲得 |
| **無料枠の制限** | 月30枚 / 抽出項目5つ / ウォーターマーク付きCSV出力 |
| **有料転換トリガー** | 処理枚数超過、抽出項目数追加、ウォーターマーク除去、API連携 |
| **想定転換率** | 5-8%（B2B SaaSの標準的な無料→有料転換率） |
| **リスク** | 無料で満足してしまう層の存在 → 無料枠を厳しめに設定して対策 |

**フリーミアム適用の判断: 適用推奨**

不定様式帳票処理は「使ってみないと精度がわからない」という性質があるため、フリープランで実際の帳票を処理して効果を実感してもらうステップが不可欠。PRJ-002以上にフリーミアムの重要度が高い。

### 4.3 PRJ-002ユーザーへのクロスセル戦略

| 施策 | 詳細 |
|------|------|
| **アプリ内プロモーション** | PRJ-002のダッシュボードにPRJ-003の案内バナーを設置 |

## 5. リスク分析

### 5.1 VLM精度への期待値と現実のギャップリスク

| リスク | 詳細 | 深刻度 | 対策 |
|--------|------|--------|------|
| **不定様式ゆえの精度ばらつき** | テンプレートがないため、レイアウトによって精度が大きく変動する可能性 | **極めて高い** | 精度のベンチマーク公開、対応帳票タイプの明確化、「確信度スコア」の表示と人間レビューフローの組み込み |
| **顧客期待値の過剰** | 「どんな帳票でも100%正確に読める」と誤解されるリスク | **高い** | マーケティングメッセージで精度の現実的な範囲を明示、「AIアシスト + 人間確認」のポジショニング |
| **ローカルAI（Ollama）の精度限界** | クラウドVLM（GPT-4o, Gemini）とローカルモデルで精度差が大きい場合 | **高い** | クラウドVLMとローカルVLMの精度比較を透明に提示、用途に応じた推奨モデルガイドの提供 |
| **複雑な帳票への対応限界** | 多段表、入れ子構造、複数ページにまたがる帳票で精度低下 | **中-高** | 対応帳票の複雑度を段階的に表示、非対応パターンの明確化 |

**最重要対策**: PRJ-003のマーケティングでは「100%自動化」ではなく「80%の作業を自動化し、残り20%を人間が確認する」というポジショニングを採用する。期待値を適切に管理することで、顧客満足度を維持する。

### 5.2 競合の参入障壁

| リスク | 詳細 | 深刻度 | 対策 |
|--------|------|--------|------|
| **大手クラウドの日本語強化** | Google/Amazon/Microsoftが日本語帳票対応を強化した場合 | **高い** | UI/UXの完成度、ローカルAI対応、PRJ-002統合という独自価値で差別化。大手が追いつく前にユーザーベースを構築 |
| **国内AI-OCR企業のVLM対応** | AI inside, Cogent Labs等がVLMベースの不定様式対応を投入 | **中-高** | 価格面の優位性維持、中小企業特化のポジション堅持、先行者優位の活用 |
| **オープンソースの台頭** | VLMベースのOCRツールがOSSとして登場する可能性 | **中** | UIの完成度とサポート体制で差別化。OSSは技術者向けであり、中小企業の非エンジニアは使えない |

### 5.3 価格競争リスク

| リスク | 詳細 | 深刻度 | 対策 |
|--------|------|--------|------|
| **クラウドAPIの価格下落** | Textract/Document AIの従量課金が更に安くなる可能性 | **中-高** | 価格ではなく「完成されたUI + 日本語最適化 + ローカルAI」の付加価値で勝負 |
| **競合の中小企業向けプラン投入** | DX Suite等がエントリープランを出してくる可能性 | **中** | フリーミアムの先行優位と、PRJ-002統合プランで対抗 |
| **AI API費用の変動** | OpenAI/Google等のAPI価格が大幅変動するリスク | **中** | ローカルAI（Ollama）対応により、API費用に依存しない運用を推奨 |

### 5.4 その他のリスク

| リスク | 詳細 | 深刻度 | 対策 |
|--------|------|--------|------|
| **PRJ-002の開発遅延の影響** | PRJ-002が未完成の場合、統合プランの提供が遅れる | **中** | PRJ-003を独立製品として先行リリースできる設計にする |
| **個人情報保護への懸念** | 帳票に含まれる個人情報の取り扱い | **中-高** | ローカルAI対応を前面に、プライバシーポリシーの整備、SOC2等の認証取得計画 |
| **VLM技術の急速な進化** | 技術変化が速く、モデル更新への追従が必要 | **中** | モデルを抽象化した設計、新モデルへの切り替えを容易にするアーキテクチャ |

## 6. SWOT分析

### Strengths（強み）
- テンプレート定義不要のUX（競合の大半はテンプレートまたは教師データが必要）
- ローカルAI（Ollama）対応によるセキュリティとコスト優位性
- PRJ-002との統合による定型+不定型ワンストッププラットフォーム
- 中小企業特化の手頃な価格設定
- 自然言語による抽出項目の指示という直感的なインターフェース

### Weaknesses（弱み）
- 導入実績ゼロからのスタート（PRJ-002と同様の課題）
- 不定様式ゆえの精度ばらつきリスク
- ローカルAIモデルの精度限界
- ブランド認知度がない
- サポート体制の構築が必要
- PRJ-002がまだリリースされていない（統合メリットの訴求が限定的）

### Opportunities（機会）
- 不定様式帳票処理は既存AI-OCRの最大の「穴」であり、未開拓市場が大きい
- VLM技術の急速な精度向上（半年単位で精度が改善）
- 中小企業のDX補助金（IT導入補助金等）の活用
- 電子帳簿保存法対応の継続需要
- 物流2024年問題に伴う業務効率化ニーズの高まり

### Threats（脅威）
- Amazon Textract QueriesやGoogle Document AIの日本語強化
- 国内AI-OCR大手のVLM対応追随
- VLM精度が顧客期待に達しない場合のレピュテーションリスク
- 価格競争の激化
- AI技術の陳腐化スピード

## 7. 推奨アクション（CEO向けサマリー）

### 即座に着手すべき事項

1. **PRJ-003のプロダクトコンセプト確定**
   - 「テンプレート不要の不定様式帳票データ抽出」を中核メッセージとして確定
   - PRJ-002との関係性（統合プラットフォーム vs 独立製品）の方針決定

2. **初期ターゲットの確定**
   - 物流・運送業を最優先ターゲットとする方針のCEO承認
   - PoC候補企業の選定着手

3. **価格戦略のCEO承認**
   - 上記SaaSモデルの価格帯についてCEOの承認を求める（マーケティング部門には価格の独断決定権限なし）
   - PRJ-002+PRJ-003統合プランの価格設定についても判断を仰ぐ

### 開発フェーズと並行して準備すべき事項

1. **精度ベンチマークの策定**
   - リサーチ部門・開発部門と連携し、主要帳票タイプごとの精度目標を設定
   - 「精度保証ライン」と「ベストエフォートライン」の定義

2. **LP（ランディングページ）の要件策定**
   - 広報Web運営部門と連携
   - PRJ-002のLPとの関係性（統合 or 個別）の決定

3. **テックブログの執筆計画**
   - 「VLMで不定様式帳票を攻略する」をテーマとした技術記事シリーズ
   - 差別化コンテンツとしてSEO効果も期待

### PRJ-002との統合に関する戦略的判断依頼

- **統合プラットフォーム化の時期**: MVP段階から統合するか、個別リリース後に統合するか
- **ブランド戦略**: PRJ-002とPRJ-003を同一ブランドで展開するか、別ブランドにするか
- **技術アーキテクチャ**: 共通基盤を使うか、別システムとして開発するか（開発部門との連携が必要）
